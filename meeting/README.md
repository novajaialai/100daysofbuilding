# Call & Meeting Recorder

Press **Super+R** to start recording; press **Super+R** again to stop and save.
Records and live-transcribes any conversation that happens on the laptop —
video meetings (Zoom/Meet/Teams), VoIP/softphone calls, or a phone call routed
into the laptop. Sibling to the `~/voice` dictation setup (Super+Z), same
faster-whisper engine.

## What it does

- Records **both sides separately**: your mic (**You**) and the system audio /
  the other end (**Them**) as two time-synced 16 kHz streams. The transcript is
  **speaker-labeled** — you can see who said what, not one mixed blob.
- **Live audio level meters** for both channels in the window, so you can *see*
  at a glance that your mic AND the far side are actually being captured.
- Live-transcribes in ~20 s chunks while the conversation is still going. Silent
  chunks are skipped (one person usually talks at a time), which keeps the
  two-stream transcription comfortably realtime even on a long call.
- On stop, appends a `## Summary` section (what it was about, key points,
  decisions, action items) — generated by the `claude` CLI, falling back to
  local ollama (`llama3.2:3b`) if offline. If both fail you get a critical
  notification and a loud note in the transcript; the transcript itself is
  always complete first.
- Transcript and audio land in `~/Meetings/<date_time>/`; `~/Meetings/latest`
  always points at the current/most recent session
  (`tail -f ~/Meetings/latest/transcript.md` works mid-call).
- After saving, the window stays open (green dot, "Saved ✓") and the transcript
  `.md` auto-opens. Pressing Super+R again starts a fresh recording.

## Built for long recordings

- **Won't sleep mid-call.** `meeting.sh` launches the recorder under
  `systemd-inhibit` (`sleep:idle:handle-lid-switch`), so the machine can't
  auto-suspend, idle off, *or* lid-suspend while recording — closing the laptop
  during a long call no longer drops it. The lock releases the moment recording
  stops.
- **Disk-full is caught before it loses data.** The watchdog monitors free space
  and fires a loud warning while there's still room (default < 2 GB), then stops
  cleanly before the disk actually fills and corrupts the recording (< 400 MB).
- **Unbounded length.** Audio is segmented straight to disk, so length is limited
  only by free space (~110 MB per hour per stream, ~220 MB/hour total at 16 kHz
  mono). If transcription ever falls behind, audio is never lost — the backlog
  count shows in the status line and is fully drained when you stop.

## Phone calls — how to get the audio in

Any call **happening on the laptop** already works out of the box, because the
other side comes through system audio and you come through the mic:

- **Softphone / VoIP** (Google Voice, Zoom Phone, Skype, a browser dialer) —
  works today, no setup. Just press Super+R.

For an **actual cellphone call**, the phone's audio has to reach the laptop, and
then you point the recorder at that source as the **far end (Them)**:

```bash
~/meeting/meeting.sh sources   # list inputs, annotated — find the phone's source
~/meeting/meeting.sh phone     # auto-pick a Bluetooth/line-in source + record
```

Two ways to bridge the phone in:

- **Bluetooth (HFP)** — pair the phone to the laptop as a headset. It then shows
  up in `meeting.sh sources` as a `bluez_input.*` (PipeWire) or `bluez_source.*`
  source. `meeting.sh phone` finds it automatically. Your voice stays on the
  laptop mic (You); the far end comes from the Bluetooth source (Them). Note HFP
  usually drops audio to ~8–16 kHz.
- **Cable / USB / line-in** — a USB audio interface or line-in dongle brings the
  phone in as an input device (best quality). Find its name in `meeting.sh
  sources` and use it.

Pick the source explicitly when you want control:

```bash
MEETING_FAR_SOURCE=bluez_input.XX_XX_XX_XX_XX_XX ~/meeting/meeting.sh   # far end = phone
MEETING_MIC_SOURCE=<some-source> ~/meeting/meeting.sh                   # if your mic isn't default
```

A bad source name **fails loudly at startup** (it won't silently record nothing).

(A call that stays entirely on the handset, never bridged to the laptop, can't be
captured — the audio has to reach the laptop somehow.) See `ROADMAP.md` for the
bigger picture.

## Fails loudly — three layers

1. **Internal watchdog** (every 2 s): recorder process alive, **both** audio
   streams actually growing, transcriber not stuck, disk not about to fill.
   Failure → window turns red, critical notification, alarm on repeat. Pure
   digital silence from the **mic** for 3 chunks (muted/unplugged?) → orange
   warning + alarm, auto-clears if audio returns.
2. **External watchdog** (`watchdog.sh`): if the supervisor process itself dies,
   you get an alarm, a critical notification, AND a zenity dialog; the orphaned
   recorder is killed so it can't record unsupervised.
3. **Startup check** (`meeting.sh`): if recording doesn't actually start within
   20 s of the keypress, same loud alarm. No silent failure modes: if you see
   the blinking dot and hear no alarm, it is recording.

## Files

- `meeting.sh` — Super+R toggle (GNOME custom keybinding `custom2`); also
  `meeting.sh sources` (list inputs) and `meeting.sh phone` (record a bridged phone)
- `supervisor.py` — GTK4 window + dual-stream ffmpeg recorder + faster-whisper
  transcriber + internal watchdog
- `watchdog.sh` — external process watchdog
- `ROADMAP.md` — where the recorder started, is now, and is going
- `meeting.log` — supervisor stderr (check here after an alarm)
- per-session `recorder.log` — ffmpeg errors; `mic/` + `sys/` hold the raw chunks

## Tuning (env vars, set in meeting.sh or globally)

- `MEETING_MODEL` — whisper model (default `base`; `small` = better accuracy,
  slower)
- `MEETING_CHUNK_SECS` — chunk length / transcription latency (default 20)
- `MEETING_SILENCE_CHUNKS` — consecutive silent mic chunks before the
  "hearing nothing" alarm (default 3; 0 disables)
- `MEETING_SILENCE_PEAK` — peak below which a chunk is treated as silence and
  skipped (default 60)
- `MEETING_SPEAKER_MIC` / `MEETING_SPEAKER_SYS` — transcript labels
  (default `You` / `Them`)
- `MEETING_FAR_SOURCE` — the far-end (Them) audio source (default: the default
  sink's `.monitor`). Set to a `bluez_input.*` / line-in / USB source to record a
  bridged phone. List names with `meeting.sh sources`.
- `MEETING_MIC_SOURCE` — your (You) audio source (default: `default`)
- `MEETING_DISK_WARN_MB` / `MEETING_DISK_FAIL_MB` — low-disk warn / hard-stop
  thresholds (default 2048 / 400)

## Notes

- Words can be clipped at chunk boundaries; the full wav chunks are kept under
  `mic/` and `sys/` in the session dir if a passage needs re-transcribing with a
  bigger model.
- Speaker separation is by audio source (your mic vs. the system output), not by
  voice — so everyone on the far side shares the **Them** label.
- Delete `mic/` + `sys/` after a session if you only want the transcript.
